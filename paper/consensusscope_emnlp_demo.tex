\begin{abstract}
AI writing-feedback systems can produce fluent comments, but teachers still
need to decide which suggestions are safe to show to students. A local grammar
edit may be acceptable, while an equally fluent suggestion may change the
student's intended meaning, add unsupported content, overcorrect a draft, or
use inappropriate student-facing language. We present ConsensusScope, an
interactive system demo for teacher-in-the-loop review routing of AI-generated
English-as-a-second-language (ESL) writing feedback. The core mechanism is a
\emph{Feedback Safety Graph}: for each feedback item, the system links the
student target span, surrounding context, AI suggestion, evidence status,
active safety dimensions, and routing decision into an auditable deploy-time
graph. ConsensusScope normalizes feedback candidates into a shared schema,
constructs these graphs without using gold labels or teacher ratings,
routes low-risk local edits separately from items that require teacher review,
and exports graph-backed reports. The demo runs without external API calls for
reproducible demonstrations, while optional live providers can write into the
same schema. We evaluate the routing layer with a packaged synthetic
stress-test suite and an offline benchmark derived from public
learner-correction corpora, including JFLEG, CoNLL-2014 annotated test data,
FCE, and W\&I+LOCNESS. Across 349,782 generated feedback candidates from public
correction data, the default policy auto-routes 32.9\% of candidates while
sending all constructed incorrect or unsafe candidates to review. A small
two-teacher diagnostic pilot further shows how teacher ratings can reveal
borderline auto-release cases and guide conservative routing updates. These
results validate the graph-backed review-routing mechanism, not classroom
learning outcomes or real LLM feedback quality.
\end{abstract}

\section{Introduction}

AI writing feedback is useful only when teachers can control what reaches a
student. Existing educational NLP work includes grammatical error correction
(GEC) shared tasks~\cite{ng-etal-2014-conll,bryant-etal-2019-bea},
learner-writing corpora such as FCE and JFLEG~\cite{yannakoudakis2011fce,
napoles-etal-2017-jfleg}, and writing-support systems that help students revise
drafts~\cite{madnani-etal-2018-writing}. Recent LLMs make feedback generation
easier, but they also increase the need for review: a fluent comment may be
locally grammatical while being pedagogically unsafe, unsupported, or too
interventionist.

ConsensusScope addresses this problem as a review-routing system rather than an
automatic essay scorer. Given an ESL draft and a set of AI-style feedback
candidates, the system asks a narrower deploy-time question: should this
specific feedback item be released directly, or should a teacher inspect it
first? We operationalize this question through a Feedback Safety Graph. Instead
of reducing feedback to one confidence score, the graph records how the target
span, AI suggestion, issue type, evidence status, and safety dimensions jointly
support an \texttt{auto\_accept}, \texttt{teacher\_review}, or
\texttt{needs\_more\_evidence} route.

This framing is intentionally practical. It supports teachers who need to
triage AI feedback before student release, educational NLP developers who need
observable failure modes, and researchers studying reliability in
human-in-the-loop feedback pipelines.

The demo contributes: (1) a Feedback Safety Graph representation for
student-facing AI writing feedback, covering local edits, meaning preservation,
content grounding, pedagogical tone, feedback specificity, and model agreement;
(2) a usable Streamlit workflow for single-essay review, batch review, feedback
comparison, teacher queueing, evaluation, and graph-backed report export; and
(3) a reproducible public learner-corpus benchmark that converts correction
gold labels into feedback-level routing diagnostics while keeping deploy-time
signals separate from offline labels.

\section{Related Work}

\paragraph{Learner writing and GEC.}
GEC benchmarks have provided standardized correction tasks and learner-writing
data, including CoNLL-2014~\cite{ng-etal-2014-conll}, BEA-2019
W\&I+LOCNESS~\cite{bryant-etal-2019-bea}, FCE~\cite{yannakoudakis2011fce}, and
JFLEG~\cite{napoles-etal-2017-jfleg}. These datasets primarily evaluate
correction quality. ConsensusScope instead uses their correction annotations as
offline evidence for a downstream question: whether a feedback-routing policy
keeps risky or incorrect suggestions in a review queue.

\paragraph{Automated writing feedback.}
Automated writing systems can provide revision support, but classroom use
depends on how learners and teachers interpret the feedback
~\cite{madnani-etal-2018-writing,ranalli2018automated}. ConsensusScope follows
this caution by avoiding a teacher-replacement claim. It is designed to make AI
feedback inspectable and triageable before it becomes student-facing.

\paragraph{LLM reliability and adjudication.}
Multi-output and judge-based methods can help select answers or evaluate
generations~\cite{wang2023selfconsistency,zheng2023judging,liu2023geval}.
However, consensus or judge confidence is not equivalent to educational safety.
Our demo treats model agreement as one observable signal, alongside issue type,
evidence status, meaning-change warnings, unsupported-claim cues, tone-safety
cues, and parse errors.

\section{Comparison with Existing Systems}

ConsensusScope differs from existing writing-support and evaluation systems in
the unit of decision it makes visible. GEC and automated writing-feedback tools
focus on producing or scoring revisions; essay-scoring systems assign
essay-level quality signals; and LLM-as-judge workflows assess generations
after they are produced. ConsensusScope instead routes each \emph{feedback
item} before it reaches a student, exposing the graph evidence behind release
or review decisions (Table~\ref{tab:comparison}).

\section{System Overview}

ConsensusScope is organized around a teacher-facing feedback-review workflow
(Figure~\ref{fig:workflow}). A teacher can paste a single anonymized ESL draft
or upload a batch CSV. The system generates no-API AI-style feedback candidates
or accepts live provider outputs in the same schema. Each feedback item is then
screened by the routing layer and displayed as either a low-risk local edit or
a teacher-review item.

\paragraph{Unified feedback schema.}
Every feedback candidate is represented with a stable item-level schema:
\texttt{feedback\_item\_id}, \texttt{essay\_id}, \texttt{target\_span},
\texttt{surrounding\_context}, \texttt{ai\_suggestion},
\texttt{ai\_rationale}, \texttt{model\_source},
\texttt{issue\_type\_predicted}, and optional \texttt{model\_agreement}. This
schema lets deterministic demo reviewers and live OpenAI-compatible providers
feed the same routing and reporting components.

\paragraph{Feedback Safety Graph.}
For each feedback item, ConsensusScope constructs a compact directed graph. The
core path is:
\begin{quote}
\small
\texttt{target span -> AI suggestion -> active safety dimension -> routing decision}.
\end{quote}
Graph nodes represent the student target span, surrounding context, AI
suggestion, optional AI rationale, predicted issue type, evidence signal,
activated safety dimensions, and route. Edges capture relations such as
\texttt{contains}, \texttt{is\_revised\_by}, \texttt{activates}, and
\texttt{justifies\_route}. The graph dimensions are local edit, meaning
preservation, content grounding, pedagogical tone, feedback specificity, and
model agreement. The exported route includes
\texttt{safety\_graph\_active\_dimensions},
\texttt{safety\_graph\_active\_signals}, \texttt{safety\_graph\_path},
\texttt{safety\_graph\_summary}, and JSON node/edge records. The graph is built
only from deploy-time observable fields; it does not use public-corpus gold
corrections, teacher ratings, or post-hoc classroom outcomes.

\paragraph{Deploy-time risk signals.}
The router uses signals observable at review time: predicted issue type,
suggestion text, target span, local context, model agreement, evidence status,
parse errors, and textual cues for meaning change, unsupported claims,
overcorrection, harsh tone, or whole-draft rewriting. It explicitly does not
use teacher ratings, public corpus gold corrections, or post-hoc labels when
making deploy-time routes.

\paragraph{Routing outputs.}
For each feedback item, the system outputs \texttt{risk\_level},
\texttt{recommended\_action}, \texttt{risk\_reasons},
\texttt{meaning\_preservation\_predicted}, \texttt{risk\_score},
\texttt{review\_confidence}, \texttt{evidence\_signal},
\texttt{review\_priority}, graph fields such as
\texttt{safety\_graph\_path} and \texttt{safety\_graph\_summary}, and a short
explanation. Low-risk grammar,
spelling, punctuation, and vocabulary edits can be routed to
\texttt{auto\_accept}; medium- and high-risk items are routed to teacher
review, more-evidence, or reject states.

\section{Target Users and Use Cases}

The primary users are writing teachers, educational NLP developers, and NLP
researchers. Teachers can inspect which AI suggestions require judgment before
student release. Developers can use the normalized schema and saved routing
records to debug feedback pipelines. Researchers can test review-routing
policies, compare risk signals, and analyze cases where generated feedback may
be grammatically fluent but educationally unsafe. The system is especially
suited for settings that need human-review routing rather than fully automatic
writing evaluation.

\section{Evaluation}

We evaluate the implemented routing layer, not a full writing-feedback
generation model. The evaluation asks whether the system separates safe local
edits from feedback that is incorrect, meaning-changing, unsupported, or
unsafe. We use three types of evidence.

\paragraph{Synthetic implementation checks.}
The packaged demo contains 15 feedback items with expected routes and 16
stress-test cases for dangerous suggestions such as thesis reversal,
whole-essay rewriting, unsupported factual claims, harsh tone, low agreement,
and parse failures. The current router matches the expected action and risk
labels on all 31 items. This is an implementation sanity check rather than a
classroom outcome study.

\paragraph{Public learner-corpus benchmark.}
We construct an offline routing benchmark from public learner-correction
corpora. For each original/corrected sentence pair, we align source and
reference tokens to create feedback-level gold edits. We then generate one
gold-derived correct feedback candidate and one or more deliberately risky
distractor candidates for each edit. The router sees only the deploy-time
feedback fields; gold labels are used afterward to compute whether incorrect
or unsafe candidates were routed to review. Table~\ref{tab:public} reports the
runs completed on directly downloadable public data.

\paragraph{Two-teacher diagnostic pilot.}
To test whether the review routes align with human judgments, we collected a
small blind pilot from two English teachers over 12 anonymized AI feedback
items. Each teacher rated correctness, meaning preservation, student
readiness, usefulness, clarity, and direct-release suitability on 1--5 scales.
The teacher ratings are used only as offline diagnostics. They are not visible
to the deploy-time router. An initial analysis identified a borderline
auto-accepted item whose wording made student release dependent on teacher
preference (``if the teacher wants''). We therefore added an observable
\texttt{teacher\_dependent} signal that routes such items to review. As shown
in Figure~\ref{fig:teacherpilot}, this update improves recall for
teacher-marked review-needed items while preserving recall for teacher-marked
unsafe items.

\paragraph{Review-routing utility.}
Table~\ref{tab:policy} summarizes the pooled policy results over 349,782
feedback candidates. The default policy accepts only low-risk items and routes
medium/high-risk items to review. It auto-routes 32.9\% of candidates while
preserving all constructed incorrect or unsafe candidates in the review queue.
The 1.000 auto accuracy should be interpreted narrowly: correct candidates are
derived from public gold corrections, and incorrect candidates are constructed
risk distractors. The number validates routing behavior under controlled
conditions; it is not a claim that real LLM feedback is always correct.

\section{Demo Scenario}

The two-and-a-half-minute demo follows a teacher workflow. The presenter opens
Review Workspace, runs Single Essay Review on a packaged anonymized ESL draft,
shows routed feedback items, and inspects Feedback Safety Graph paths for
auto-accepted local edits and teacher-review items. The demo then processes a
batch CSV, opens AI Feedback Comparison, selects a high-risk item in Teacher
Queue, and exports a Markdown report. The Effectiveness Evaluation page shows
the synthetic stress-test metrics and the public learner-corpus aggregate
table.

The public benchmark can be reproduced with
\texttt{scripts/run\_public\_gec\_benchmark.py}. JFLEG can be downloaded
automatically by the script. CoNLL-2014 annotated test data and BEA-2019 FCE
and W\&I+LOCNESS data are downloaded from their official release pages. Raw
corpora and per-item derived outputs are not committed to the repository;
aggregate result tables are included for inspection.

\section{Ethics and Limitations}

ConsensusScope is a teacher-support tool, not an automatic essay scorer or
teacher replacement. Users should upload only anonymized student writing; names,
student IDs, emails, class identifiers, and other personally identifiable
information should be removed before use. The current benchmark validates
graph-backed review-routing behavior with public correction labels and
constructed distractors. It does not measure student learning, teacher
satisfaction, time-on-task, or real LLM feedback quality. The two-teacher
pilot is preliminary and covers only 12 feedback items; it should be read as a
diagnostic check for routing behavior, not as a classroom user study. Future
work should expand the teacher sample, add authentic anonymized classroom
data, and keep deploy-time graph signals separate from offline diagnostic
ratings.

\section{Conclusion}

ConsensusScope demonstrates a practical Feedback Safety Graph layer for
AI-generated ESL writing feedback. By normalizing feedback candidates,
constructing item-level safety graphs, and routing uncertain or unsafe
suggestions into a teacher queue, the system turns AI feedback from a final
rewrite into an auditable educational workflow.
