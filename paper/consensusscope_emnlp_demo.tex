\begin{abstract}
Large language models can provide fluent writing feedback, but in educational
settings the most useful question is often not whether a model can rewrite a
sentence, but whether a suggested change is safe to accept without a teacher.
This is especially important for English-as-a-second-language comparative
literature writing, where low-risk grammar edits coexist with factual claims
about authors, dates, genres and characters, and with interpretation-changing
argument revisions. We present ConsensusScope, an interactive system demo for
knowledge-grounded multi-LLM feedback adjudication. The system retrieves a
curated literary knowledge graph, asks deterministic or live OpenAI-compatible
reviewers to produce feedback in a unified schema, and routes suggestions
through a transparent adjudication layer. Low-risk local language edits can be
auto-accepted, while literary facts and meaning-changing revisions are routed
to a teacher-facing review queue with evidence and rationale. The current
release contains 319 curated knowledge triples over 30 commonly taught literary
works, a 30-case ESL feedback-routing benchmark, and a live validation run with
DeepSeek, Qwen, GLM and Kimi. ConsensusScope also retains general multi-model
QA audit pages as an auxiliary reliability module, but the main demo claim is
review-routing for educational feedback rather than fully automatic essay
grading.
\end{abstract}

\section{Introduction}

Automated writing feedback has long been studied in educational NLP, including
grammatical error correction shared tasks~\cite{ng-etal-2014-conll,
bryant-etal-2019-bea}, automated writing evaluation systems
~\cite{madnani-etal-2018-writing}, and work on how learners use automated
written corrective feedback~\cite{ranalli2018automated}. Recent LLMs make such
feedback easier to generate, but they also make a familiar classroom problem
more acute: a fluent suggestion may be locally grammatical while being
pedagogically unsafe, factually wrong, or overly interventionist.

This issue is visible in ESL comparative literature essays. A model may safely
correct ``Both novels shows'' to ``Both novels show.'' The same model, however,
may also rewrite a student's thesis, assert that \textit{Jane Eyre} was written
by Jane Austen, or erase an interpretive ambiguity in \textit{The Turn of the
Screw}. In this setting, treating all feedback as equally safe is inappropriate.
The system should distinguish local language edits from suggestions that
require expert verification.

ConsensusScope addresses this problem as a review-routing layer for
multi-LLM feedback. Instead of presenting a single model revision as final, the
system collects feedback suggestions from multiple reviewers, grounds relevant
claims in a small literary knowledge graph, estimates meaning-change risk, and
separates auto-accepted edits from teacher-review items. The goal is not to
replace teachers or grade full essays automatically. The goal is to make AI
feedback auditable before it changes a student's text.

The demo makes three contributions. First, it reframes multi-model
adjudication as an applied educational NLP workflow for ESL comparative
literature feedback. Second, it introduces a transparent knowledge-grounded
feedback schema and teacher-review routing policy. Third, it provides an
installable Streamlit interface with no-API deterministic reviewers, live
OpenAI-compatible reviewers, saved benchmark records, risk dashboards and
report export.

\section{Related Work}

\paragraph{Learner writing feedback.}
Grammatical error correction has developed through shared tasks and learner
corpora~\cite{ng-etal-2014-conll,bryant-etal-2019-bea}. Automated writing
systems also support higher-level feedback and self-regulated revision
~\cite{madnani-etal-2018-writing}. At the same time, automated written
corrective feedback can be uneven across error types and may require careful
classroom interpretation~\cite{ranalli2018automated}. ConsensusScope follows
this caution: it does not assume that generated feedback should be directly
applied.

\paragraph{Multi-LLM adjudication.}
Multi-output methods such as self-consistency~\cite{wang2023selfconsistency}
and debate~\cite{du2023debate} can improve final-answer selection, while
LLM-as-a-judge methods provide scalable evaluation~\cite{zheng2023judging,
liu2023geval}. However, agreement and judge confidence are not the same as
safety. ConsensusScope therefore treats consensus as one signal among others
and exposes the routing decision to the user.

\paragraph{Factuality and evidence.}
Prior work on hallucination, factuality and uncertainty shows that fluent
generations can be unsupported or miscalibrated~\cite{manakul2023selfcheckgpt,
min2023factscore,kadavath2022language}. Our system applies this lesson to
writing feedback: suggestions that involve literary facts or interpretation
should carry evidence and remain reviewable.

\section{System Overview}

Figure~\ref{fig:workflow} summarizes the demo workflow. A user enters an ESL
comparative literature essay excerpt or selects a prepared example. The system
retrieves relevant knowledge triples, generates feedback suggestions, normalizes
them into a shared schema, adjudicates them by risk, and presents a teacher
review queue. The current Streamlit app contains eight pages: Home/System
Overview, Live Question Mode, Sample Audit Mode, Adjudication Comparison, Risk
Dashboard, Model Reliability Dashboard, Case Explorer and Report Export. The
ESL feedback workflow is the default path in Live Question Mode; the original
multi-model QA audit workflow remains available as a secondary mode.

\paragraph{Knowledge graph.}
The current release includes a curated task-specific literary knowledge graph,
not a complete open-domain literary database. It contains 319 triples over 30
commonly taught works. Relations cover authorship, publication date, literary
form, genre, central characters, themes and title aliases. The graph is used to
support inspectable feedback, for example correcting an authorship claim or
flagging a genre mismatch.

\paragraph{Feedback generation.}
ConsensusScope supports two generation modes. The no-API mode uses deterministic
local reviewers so the demo can run in a classroom or reviewer environment
without external calls. The live mode calls OpenAI-compatible providers such as
DeepSeek, Qwen, GLM and Kimi. Every reviewer returns the same schema:
\texttt{span}, \texttt{issue\_type}, \texttt{suggestion}, \texttt{rationale},
\texttt{confidence}, \texttt{knowledge\_evidence} and
\texttt{meaning\_change\_risk}. This uniform schema makes model outputs
auditable even when providers differ.

\paragraph{Adjudication.}
The adjudication layer groups suggestions by issue span and type, then assigns
each decision to \texttt{auto\_accept} or \texttt{teacher\_review}. Local
grammar and word-choice edits with low meaning-change risk may be auto-accepted.
Literary facts supported by the knowledge graph are routed to teacher review,
because evidence still needs pedagogical inspection. Argument, thesis and
interpretation suggestions are also routed to review because they may change
the student's intended meaning.

\section{Target Users and Use Cases}

The primary users are NLP researchers and educational-technology developers who
need to inspect feedback pipelines, and teachers who need to audit AI-generated
feedback before it reaches students. Researchers can use the system to compare
deterministic and live feedback generation, inspect evidence usage and identify
failure modes. Developers can use the unified schema and saved records to test
review-routing policies. Teachers can use the exported report to see which
edits were auto-accepted and which require manual judgment.

\paragraph{Auxiliary reliability module.}
The repository also retains saved multi-model QA traces over TruthfulQA,
FEVER and CommonsenseQA~\cite{lin2022truthfulqa,thorne2018fever,
talmor2019commonsenseqa}. These pages support sample audit, risk dashboarding
and comparison among majority vote, a fixed judge and rule-based dynamic
adjudication. We keep this module as background infrastructure, but do not use
mixed-task macro-F1 as the paper's headline metric.

\section{Demo Scenario}

The two-and-a-half-minute demo begins on the Home/System Overview page, then
opens Live Question Mode with the ESL literary feedback option. The presenter
loads the \textit{Frankenstein} and \textit{Jane Eyre} example, runs
knowledge-grounded feedback, and shows the split between auto-accepted grammar
edits and teacher-review items. The Knowledge Evidence tab displays author and
publication-year triples. The Adjudication Trace tab shows why factual and
argument-level suggestions were not automatically applied. The demo then
briefly visits Sample Audit Mode and Risk Dashboard to show that the same
observability framework also supports saved multi-model QA traces. Finally,
Report Export downloads a Markdown feedback report.

\section{Ethics and Limitations}

ConsensusScope is a teacher-support and audit tool, not an automatic grading
system. The curated knowledge graph is intentionally small and should not be
described as complete. Knowledge-supported feedback can still be inappropriate
for a student's local argument, so factual and interpretation-changing
suggestions remain routed to teacher review. Live API behavior can change with
provider updates, and saved validation records are included to make the demo
inspectable without exposing API keys. Future work should add teacher
annotations, classroom usability studies and a larger literary knowledge base.

\section{Conclusion}

ConsensusScope demonstrates a practical review-routing layer for
knowledge-grounded multi-LLM writing feedback. By separating low-risk local
language edits from factual and interpretation-changing suggestions, the system
turns multi-model agreement into an auditable classroom workflow rather than a
black-box final rewrite.
